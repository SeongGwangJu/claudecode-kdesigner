# AI-Slop Vocabulary — 회피 어휘·질문 패턴 카탈로그

> `aesthetic-guard` SKILL이 *부정 패턴 식별*과 *디자이너 묻기*에 참조하는 카탈로그. 본체에 박지 않은 이유: 데이터(어휘 리스트)지 행동 분기가 아니어서 — 어휘 추가/수정이 SKILL 본문 변경 없이 가능해야 함.

---

## 1. 시각 회피 리스트

### 1.1 타이포
- 흔한 폰트 단독: Inter, Roboto, Arial, Helvetica, system-ui, Open Sans, Lato
- 무지개 그라디언트 텍스트 (특히 보라→파랑 / 분홍→보라)
- 디자이너 프로필 명시 폰트 우선, 없으면 의도적 display + 다른 결의 body 페어링

### 1.2 색
- 보라(`#6366f1`/`#7c3aed` 류) 그라디언트 + 흰 배경
- 모든 카드가 똑같이 옅은 회색 그림자
- 강조 없이 균등 분포된 8-10가지 색

### 1.3 레이아웃
- 카드 그리드 — 같은 크기 박스가 정렬된 채 모든 정보가 같은 무게
- 화면 맨 위 큰 글씨 + 부제 + CTA 버튼 한 줄, 같은 구도의 반복
- 정확히 정렬된 대칭 — 의도적 비대칭/오버랩이 더 살아있음

### 1.4 아바타·인물·로고 — 이니셜 박스/원형
- **이니셜 박스/원형 아바타** ("Kim" → 회색 원에 "K", "박지우" → 검정 사각형에 "박")
- **이니셜 박스/원형 브랜드 로고** (검정 사각형 + 흰 'S', 색 원 + 'K' 같은 *최소 결정만 한 로고*)
- 회색 사람 실루엣 아이콘 / 단색 원 + 무작위 색

→ `aesthetic-guard` 본체 §이니셜 박스 회피 — 적극 분기 발동.

### 1.5 더미 데이터
- "Lorem ipsum"
- "John Doe", "user@example.com", "Sample Title"
- 의미 없는 숫자 (`123`, `999`, `0`) — 한국 서비스면 한국어 실명·실제 카피가 화면을 살림

## 2. 한국어 카피 슬롭 — 의미 없는 형용사·과장어

- "혁신적인", "원활한", "차세대", "게임 체인저", "최첨단", "획기적인"
- 과장 부사 + 일반 동사 조합 ("완벽하게 관리합니다", "효율적으로 처리합니다")
- → 구체 명사·구체 수치로 ("혁신적인 관리 시스템" → "47,200건 등록·작년 +32%")

---

## 3. 제목·부제 동어반복 — 가장 흔한 한국 어드민 슬롭

- **디폴트는 *부제 없음***. 헤더 자리에 부제 슬롯이 *있다고 채워야 한다*는 충동에 저항하라. 부제는 *제목에 없는 새 정보가 있을 때만* 추가하는 *예외*다 — *기본 동작*이 아니다.
- 부제가 제목을 *재진술*만 하는 패턴: "공지사항 상세" + "공지사항 내용", "사용자 관리" + "사용자를 관리합니다", "설정" + "설정 페이지"
- 부제 추가 허용 케이스: 메타(작성자·일시), 범위(담당 시설 목록), 강조 카피(가치 한 줄). 그 외엔 제거.

**예시**:
- ✅ "공지사항 상세" + "2026.04.24 · 슈퍼관리자"
- ✅ "대시보드" + "담당 시설 5곳"
- ❌ "공지사항 상세" + "공지사항 내용"
- ❌ "사용자 관리" + "효율적으로 관리합니다"

## 4. 부족한 차원 묻기 — 질문 패턴

이 플러그인 페르소나는 Claude Code 기능을 잘 모르는 디자이너. 미학 결정을 *암묵적으로 평균에 맞추면* AI slop 직행 — 그래서 부족 차원은 *적극적으로 묻는다*. 단, 한 번에 1개만, 자연스럽게.

### 4.1 묻기 전 — 코드 추출 우선 (불필요한 질문 차단)
1. `globals.css`/`tailwind.config.*` → 색·폰트·둥글기·간격 추출
2. `components/` → 기존 스타일 결 추출 (radius·shadow·typography 어떤 결?)
3. `~/.claude/CLAUDE.md` §디자이너 프로필 (전역 켜져 있으면)
4. `CLAUDE.project.md` §디자인 시스템 슬롯들

위 4단계로도 *비어있는* 차원만 디자이너에게 묻는다.

### 4.2 좋은 질문 패턴 (디자이너 어휘)
- 톤이 비어있을 때: "이 화면 *어떤 분위기*로 가져가시고 싶어요? 한 단어라도 좋아요. (예: 미니멀 / 에디토리얼 / 플레이풀 / 럭셔리 / ...)"
- 레퍼런스가 비어있을 때: "참고할 만한 화면이 1개 있으면 보여주세요. URL·이미지·서비스 이름 다 좋아요."
- 철학이 비어있고 톤만으론 부족할 때: "이 화면이 *제일 우선해야 할 한 가지*가 뭘까요? 정보 빠르게 / 분위기 / 신뢰감 / 재미 중에서요."

### 4.3 안 좋은 질문 패턴 (회피)
- "이 프로젝트의 디자인 철학은?" — 너무 열림
- "색·간격·둥글기·타이포 다 정해주세요" — 한 번에 너무 많은 결정
- "어떤 폰트 쓸까요?" — 디자이너 어휘 X (어휘는 *결*·*분위기*·*톤*)
- "피하고 싶은 디자인 있으세요?" — *사전 인터뷰* X. 회피는 사후 누적이 정확함 (`feedback-curator` Skill 책임).

### 4.4 판단 가이드
- 글로벌 프로필이 채워져 있으면 그대로 가져다 쓰고 *묻지 X*. 디자이너가 "이번엔 다른 톤" 명시할 때만 재질문.
- `import-existing`이 추출한 토큰·폰트가 있으면 그게 답의 일부. 거기서 *부족한 차원*만.
- 결정 피로 글로벌 §6 — 한 사이클 질문 3회 상한. 다른 Skill이 이미 묻고 있으면 침묵.
- 모든 객관식엔 "**잘 모르겠어요 / 추천해주세요**" 옵션 필수.

## 5. 변형 탐색 분기 — 묻는 대신 *보여주기* (초기 화면 1개 한정)

방향성이 *언어로 표현하기 모호*한 경우, 묻는 것보다 *2-3개 atomic 변형을 코드에 박은 채 보여주는* 분기가 더 정확할 때가 있다 — *디자이너 어휘는 종종 말이 아니라 보는 것*.

### 5.1 조건
- *초기 화면 1개* 한정. 두 번째 화면부터 변형 출력 X(피로 ↑).
- 직전 단계(프로필·코드 추출·레퍼런스 흡수)에서 *지배 톤은 정해졌고* 1-2 차원만 모호한 경우.
- 또는 디자이너가 명시적으로 "변형 보여달라" 요청한 경우.

### 5.2 변형 구성
- **A — 안전한 결**: 기존 코드·프로필·레퍼런스에서 *가장 자연스러운 추론*
- **B — 대조 결**: 정반대 방향 (A가 미니멀이면 B는 맥시멀, A가 차분이면 B는 강렬)
- **C — 사이 또는 novel 결**: A·B 사이 또는 *디자이너가 안 떠올렸을 결* (실험)

각 변형엔 *결 한 단어*만 라벨로 ("정제된", "강렬한", "에디토리얼"). 디자이너가 고르면 그 결로 commit, 나머지 코드 즉시 삭제.

**묻기 + 보여주기 동시 X** — 둘 중 하나만(중복은 결정 피로).

## 6. 톤 카탈로그 (좁게 잡지 않기)

방향 결정 시 후보를 좁게 잡지 X — *11개 이상 결*에서 고른다:

미니멀 / 맥시멀리스트 / 에디토리얼·매거진 / 브루탈·거친 / 레트로·퓨처리스틱 / 유기체·자연 / 럭셔리·정제 / 플레이풀·장난기 / 아르데코·기하 / 파스텔·소프트 / 산업적·실용

**방향 결정 핵심**: 어중간한 평균 X. 미니멀이면 *제대로* 미니멀(여백·정밀 타이포), 맥시멀이면 *제대로* 맥시멀(정교한 모션·레이어).
