> 2. **파일이 존재하면** — 다른 어떤 응답도 만들기 전에 §6 마이그레이션 분기를 처음부터 끝까지 따라간다. 절대 만들지 않는 응답: "이미 시작되어 있어요 — `/kdesigner:디자인끄기`로 정리한 뒤 다시 부르세요" 같은 *분기 차단* 응답. 막지 말고 §6.4 패턴 중 하나로 *처리 결과*를 말한다.
> 3. **파일이 없으면** — §0부터 정상 흐름.

# 디자인초기설정

## 언제 쓰나
디자이너가 *처음으로* 이 프로젝트(또는 처음으로 이 컴퓨터)에서 디자이너 모드를 켤 때. 시작 *의식*이라 1회성, 이후엔 자연어("새로 만들고 싶어"/"보여줘"/"저장해줘")로 흘러간다.

## 동작

### 0. 사전 도구 확인 (가장 먼저)

디자이너 모드 작동에 필요한 도구가 다 있는지 먼저 확인한다. 다 있으면 짧은 한 줄("도구 확인 완료") 후 §1로 진행. 빠진 게 있으면 *어떻게 깔지* 한국어로 친절히 안내하고 흐름을 *멈춘다* — 사용자가 설치/설정 후 `/kdesigner:디자인초기설정`을 다시 호출하면 이어서 시작한다.

#### 0.1 검사 항목 (한 번에 다 검사 후 결과 정리)

먼저 `uname -s`로 OS 판별(`Darwin` = macOS, 그 외 = Windows/Linux). 안내 문구가 OS에 따라 갈라진다.

| 항목 | 명령 | 분류 |
|---|---|---|
| 화면 엔진 (`Node.js`) | `node --version` | 필수 (없으면 작업 자체 불가) |
| 작업 시점 저장 도구 (`git`) | `git --version` | 필수 |
| 사용자 표시 정보 | `git config --global user.name` + `git config --global user.email` | 필수 |
| 도구 설치 도우미 (`Homebrew`) | `brew --version` | macOS 한정 권장 (있으면 다른 도구 설치가 한 줄로 끝) |
| 팀 공유 공간 도구 (`gh`) | `gh --version` | 선택 (인계 단계에서 필요) |
| 팀 공유 로그인 | `gh auth status` | 선택 |

검사 결과는 *한꺼번에* 모은다 — 사용자가 한 번에 무엇을 깔지 알 수 있게(깔다 다시 호출, 또 깔다 다시 호출 흐름 방지).

#### 0.2 모든 도구 OK인 경우

응답 본문 맨 위에 진행 단계 가시화 한 줄 노출 — 메시지는 `plugin/commands/references/tool-check-messages.md` §모든 도구 OK인 경우 그대로. → §1으로 그대로 진행.

#### 0.3 필수 도구 누락 → 흐름 정지 + 친화 안내

영어 에러 한 줄도 노출 X. *해당하는 항목만* 묶어 한 응답에 함께 노출.

**OS·brew 분기 결정**:
- `uname -s` = `Darwin` (macOS) + `brew --version` 성공 → brew 경로 메시지 우선
- `uname -s` = `Darwin` + brew 없음 → macOS brew 없음 fallback 메시지
- Windows / Linux → 공식 사이트 fallback 메시지

각 도구별 *그대로 노출할* 한국어 안내 메시지는 `plugin/commands/references/tool-check-messages.md` §필수 도구 누락 참조:
- `Homebrew` (macOS 권장)
- `Node.js` (필수, OS·brew 분기)
- `git` (필수, OS·brew 분기 — Windows는 공식 사이트)
- `git` 사용자 정보 미설정 (`AskUserQuestion`로 이름·이메일 받아 `git config --global` 실행. 거부 시 흐름 중단)

#### 0.4 선택 도구 누락 → 안내만, 흐름 진행

`gh` 미설치 / `gh auth status` 실패 / 로그인 안 된 경우 — `plugin/commands/references/tool-check-messages.md` §선택 도구 누락 참조. 한 줄만 본문 위에 노출 후 *필수 도구가 다 있다는 전제로* §1로 그대로 진행.

#### 0.5 흐름 정지 시 응답 끝 안내

필수 도구 누락으로 흐름을 멈춘 경우, `plugin/commands/references/tool-check-messages.md` §흐름 정지 시 응답 끝 안내를 응답 마지막에 그대로 노출. 다음 행동 1개 제안(글로벌 §7) 자리는 이 안내가 대체.

### 1. 사전 점검
다음을 `Read`/`Bash test`로 빠르게 확인하고, 분기 결정에만 쓴다(중간 응답에는 노출 X):

| 항목 | 확인 |
|---|---|
| 현재 디렉토리에 `CLAUDE.project.md` 존재 | 이미 켜져 있음 → §6 "이미 시작됨 — 마이그레이션 분기"로 |
| `~/.claude/CLAUDE.md` 존재 + 디자이너 페르소나 마커(`<!-- kd:designer-mode:start -->`) | 전역 이미 켜져 있음 → §3 스킵, §6의 *전역 슬롯 매칭*도 함께 |
| `./CLAUDE.project.md.kd-archived-*` 발견 (하나 이상) + `./CLAUDE.project.md` 부재 | 이전 세션 작업 흔적 → §1.5 복원 분기 |
| 현재 디렉토리에 `package.json` 또는 `components/` | 기존 코드 → §5에서 `import-existing` 권유 |
| 완전 빈 디렉토리(또는 `.git`만) | §5에서 `new-service` 권유 |

**중요**: "이미 시작됨" 상태라도 흐름을 *막지 않는다*. §6에서 *최신 템플릿과의 diff*를 점검해 빠진 슬롯만 자동 추가(idempotent). 신규 설치자는 §6 분기에 진입하지 않으므로 영향 0.

### 1.5 archived 발견 시 복원 분기 (선택)

`./CLAUDE.project.md`은 *없는데* `./CLAUDE.project.md.kd-archived-*`가 하나 이상 있으면 — `/kdesigner:디자인끄기`로 끄고 다시 켜는 흐름. 이전 작업의 컴포넌트 인덱스·디자인 토큰을 살릴지 묻는다.

`AskUserQuestion`:

> 이 폴더에 *이전에 디자이너 모드로 작업한 흔적*이 있어요(`CLAUDE.project.md.kd-archived-...`). 그때 정리해뒀던 디자인 약속·컴포넌트 정리를 *이어서* 쓸 수 있어요.
>
> - **이어서 쓰기** (이전 작업 내용 그대로 살리기, 다시 분석 필요 X)
> - **새로 시작** (이전 흔적은 보관해둔 채 비어있는 상태로 새로 시작)
> - **잘 모르겠어요** → 기본값: 이어서 쓰기

선택 결과대로:
- **이어서 쓰기**:
  - `ls -t ./CLAUDE.project.md.kd-archived-* | head -1` 로 가장 최근 archive 추출
  - `mv` 로 그 파일을 `./CLAUDE.project.md`로 복원 (cp 아님 — 한 archive를 두 자리에 두지 않음)
  - 그 후 §2.2 import 라인 처리는 *복원된 파일 기준으로 그대로* 진행 (마커 검증 통과)
  - §2.1 템플릿 생성은 *스킵* (이미 `CLAUDE.project.md` 존재)
- **새로 시작**:
  - archive 파일은 *손대지 않음* (보관)
  - §2.1부터 그대로 진행 (템플릿으로 새 `CLAUDE.project.md` 생성)

응답에 복원 시 한 줄 노출:
> 이전 작업 그대로 이어서 시작해요 — `CLAUDE.project.md.kd-archived-YYYYMMDD-HHmm`을 다시 살렸어요.

### 2. 프로젝트 단위 활성화 (기본, 묻지 않음)
프로젝트 단위는 위험이 거의 없음 — 사용자 디렉토리 *내부*만 건드림. **묻지 않고** 진행.

§1.5에서 *이어서 쓰기*로 복원했으면 §2.1은 스킵하고 §2.2부터.

#### 2.1 `CLAUDE.project.md` 생성

`plugin/templates/CLAUDE.project.md`(이 플러그인의 템플릿) 내용을 그대로 가져와 현재 디렉토리 루트에 `CLAUDE.project.md`로 생성:

- `Read` 템플릿 → `Write` 프로젝트 루트에 (`./CLAUDE.project.md`)
- `{{서비스명}}`은 폴더 이름(`basename "$PWD"`)으로 채움
- 나머지 `{{...}}` placeholder는 *그대로 둔다* — `new-service`/`import-existing`이 이후 채울 영역
- 이미 존재하면 덮어쓰지 *않고* §6 "이미 시작됨" 분기

#### 2.2 `./CLAUDE.md`에 import 라인 추가 (자동 로드 보장)

`CLAUDE.project.md`는 *비표준 이름*이라 새 대화 시작 시 자동 로드되지 않는다 — 자동 로드되는 파일은 `./CLAUDE.md`, `./.claude/CLAUDE.md`, `./CLAUDE.local.md`, `~/.claude/CLAUDE.md`. 따라서 자동 로드되는 `./CLAUDE.md`에 공식 `@<path>` import 문법으로 한 줄 박아 *새 대화에서도 디자인 약속(토큰·컴포넌트 인덱스·페르소나 요약)이 그대로 이어지게* 한다.

처리 절차:

1. **`./CLAUDE.md` 존재 여부 확인** — `Read` 또는 `test -f ./CLAUDE.md`.

2-a. **없으면** (새로 생성):
   - `touch ./.kd-no-prior-claude-md` — 빈 마커 파일. `/kdesigner:디자인끄기`가 *우리 플러그인이 새로 만든 파일인지* 판단해서 통째 제거할 수 있게.
   - `Write ./CLAUDE.md`로 다음 내용만 박는다:
     ```
     <!-- kd:designer-mode:start -->
     @CLAUDE.project.md
     <!-- kd:designer-mode:end -->
     ```

2-b. **있으면** (기존 파일에 마커 격리해 추가):
   - **백업** — `cp ./CLAUDE.md ./CLAUDE.md.kd-backup-<YYYYMMDD-HHmm>` (timestamp는 `date +%Y%m%d-%H%M`). `/kdesigner:디자인끄기`에서 *원본 그대로* 복원할 수 있게.
   - **append** — 기존 파일 끝에 빈 줄 하나 + 다음 마커 격리 블록 append:
     ```
     
     <!-- kd:designer-mode:start -->
     @CLAUDE.project.md
     <!-- kd:designer-mode:end -->
     ```

3. **마커 검증** — append/생성 후 `Read ./CLAUDE.md`로 다시 읽어 시작 마커(`<!-- kd:designer-mode:start -->`)와 끝 마커(`<!-- kd:designer-mode:end -->`)가 *정확히 1쌍*인지 확인. 0쌍 또는 2쌍 이상이면 사용자에게 응답 끝에 경고 1줄 노출 ("`./CLAUDE.md`에 디자이너 마커가 정상이 아닌 상태예요 — 한 번 확인해주세요").

마커 사이는 *오직 `@CLAUDE.project.md` 한 줄만*. 디자인 시스템·페르소나 본문은 박지 않는다 — 본문은 `CLAUDE.project.md`에 있고, `@import`이 자동 로드시킨다.

### 3. 전역(user-level) 활성화 — 1회 동의 (권장 표시)

전역 `~/.claude/CLAUDE.md`에 디자이너 페르소나 + 프로필 슬롯을 박는다. 강제 X — 1회 동의받되, *왜 권장인지*를 옵션 카피에 명시.

`AskUserQuestion`:

> 디자이너 모드를 **이 컴퓨터 전체**에서 켜둘까요?
>
> **켜두면 뭐가 좋나요**: 다른 프로젝트로 가도 Claude Code가 *처음부터* 한국어로 답하고, 흔한 AI 화면(보라 그라디언트·Inter 폰트·이니셜 아바타 같은 클리셰)을 자동으로 회피하고, 디자이너 친화 톤으로 동작해요. *이 컴퓨터의 Claude Code가 디자이너용 도구로 변하는* 거예요.
>
> **디자이너 프로필을 한 번만 답해두면**: 매 프로젝트마다 "어떤 결로 갈까요?" 묻지 않고, *내 평소 결*에서 시작해요.
>
> **무엇이 바뀌나요**: `~/.claude/CLAUDE.md` 끝에 디자이너 페르소나 + 프로필 슬롯이 추가돼요(원본은 그대로 백업).
> **언제 되돌리나요**: `/kdesigner:디자인끄기` 한 번이면 원본 그대로 복원돼요.
>
> - **이 컴퓨터 전체에서 켜기 (권장)** — 모든 프로젝트에서 자동 디자이너 모드 + 디자이너 프로필 인터뷰 2질문
> - **이 프로젝트에서만 켜기** — 다른 프로젝트는 영향 X (프로필 인터뷰도 스킵)
> - **잘 모르겠어요** → 기본값: 이 컴퓨터 전체에서 켜기

선택 결과대로:

#### 3-A. "이 컴퓨터 전체에서 켜기" 또는 "잘 모르겠어요" 선택 시 (기본 분기)
1. **백업** — `~/.claude/CLAUDE.md`가 있으면 `~/.claude/CLAUDE.md.kd-backup-<YYYYMMDD-HHmm>`로 `cp`. 없으면 빈 백업 마커(`~/.claude/.kd-no-prior-claude-md`) 생성.
2. **추가** — `plugin/templates/CLAUDE.user.md` 내용을 `~/.claude/CLAUDE.md` 끝에 *append*. 시작·끝 마커로 격리:
   ```
   <!-- kd:designer-mode:start -->
   ... (templates/CLAUDE.user.md 내용)
   <!-- kd:designer-mode:end -->
   ```
   `/kdesigner:디자인끄기`가 이 마커 사이만 잘라 복원할 수 있게.
3. 백업 경로를 응답에 1줄로 노출 — "원본은 `~/.claude/CLAUDE.md.kd-backup-YYYYMMDD-HHmm`에 보관해뒀어요".

#### 3-A.1 디자이너 프로필 인터뷰 (전역 켤 때만 — 자유 응답 → 객관식 순)

전역에 박힌 `CLAUDE.user.md`의 §디자이너 프로필 슬롯을 채우는 인터뷰. *한 번 답해두면 새 프로젝트마다 같은 질문을 반복하지 않음*. `aesthetic-guard` Skill이 코드 출력 직전 이 슬롯을 읽어 한 방향에 commit.

##### A. 자유 응답 단계 — `AskUserQuestion` 아님, 평문 1회

디자인은 주관적 영역이 많아 객관식 칸으로는 디자이너 평소 결을 다 담기 어려움. *객관식 묻기 전에* 자유롭게 풀어 답할 기회를 먼저 준다 — 디자이너 어휘는 종종 *말이 아니라 보는 것*이라 글로 짧게 잘라 묻는 형식보다 *한 번에 자유롭게*가 더 정확. 평문 메시지로 다음 안내:

> 본격 시작 전에, 평소 본인 결을 **한 번에 자유롭게** 알려주시면 더 정확해요. 아래 항목 중 *답하고 싶은 것만* 자유롭게 풀어주세요 — 객관식 칸이 아니라 자유 텍스트예요.
>
> *자유롭게 답해도 좋은 항목*:
> - 최근 *눈이 가서 저장해둔* 디자인·프로덕트 (1-3개, 이름·URL 어느 거든) — *왜 좋았는지*도 한 단어라도 함께 (예: "절제된 모노 결", "한글 타이포 살아있음")
> - 본인의 *최근 작업이나 포트폴리오 URL* (있다면)
> - 평소 본인 결을 *한 단어*로 (예: 단정한·따뜻한·날카로운)
> - 이번 작업이 *어떤 분위기*면 좋을지
> - *절대 안 됐으면* 하는 결 (있다면)
>
> *파일·이미지·문서 첨부도 OK* — 어디든 편한 방법으로:
> - 이 대화창에 **끌어다 놓기** (이미지·PDF 직접 첨부)
> - 이미지 **복사·붙여넣기**
> - 프로젝트 폴더 어디에 파일 두고 *"참고로 `<경로>/<파일명>` 봐주세요"*라고 한 줄
> - 외부 *공개 URL*도 OK (블로그 글·디자인 갤러리 등)
> - **Figma 링크는 그대로 붙여넣으면 인증 때문에 못 봐요** — Figma에서 *PNG/SVG로 export*(우클릭 → Export 또는 File → Export)해서 이 대화창에 첨부 부탁드려요. 공개 페이지가 아닌 한 링크만으론 시각 콘텐츠 추출 불가(Figma MCP 별도 설치 시에만 가능 — 일반 디자이너 환경에선 export 우회가 표준).
>
> 답 안 하셔도 됩니다 — *"다음으로"* 또는 *"객관식으로"*라고 답하시면 객관식 단계로 넘어가요.

**자유 응답 처리**:
- 디자이너가 *답한 항목만* 파싱해 §디자이너 프로필 슬롯에 박는다 — 레퍼런스 → `fav-references`(원문 그대로), 포트폴리오 URL → `portfolio`, 톤 단어 → `tone-extracted`, 회피 → `avoidance`.
- 첨부된 이미지·PDF가 있으면 `Read`로 흡수 — 색·폰트·구도 추출해 슬롯에 *원문 + 추출 요약 한 줄* 박기 (예: "*Pretendard 페이지 PDF 첨부 → 한글 타이포가 살아있는 결*"). 디자이너가 *어떤 추론이 됐는지* 볼 수 있어야 함.
- 첨부 파일 경로를 받았으면 `Read`로 직접 열기 — 이미지·PDF 지원. **PPTX/DOCX/HWPX는 Read 직접 X** — *"이 형식은 아직 직접 못 읽어요. 핵심 내용을 텍스트로 알려주시거나, PDF로 export 부탁드려요"* 한 줄 안내 후 자유 응답 그대로 진행 (강제 X).
- *답하지 않은 슬롯*만 §B 객관식 분기로. 모든 슬롯이 자유 응답으로 채워졌으면 §B 전체 스킵.
- **`AskUserQuestion` 사용 X** — 이 단계의 본질은 *객관식 한계 넘기*. 도구가 객관식이면 의미 없음. 자유 응답은 평문 텍스트로.

##### B. 객관식 분기 — 자유 응답에서 빈 슬롯만

자유 응답에서 *빈 슬롯이 0개*면 §B 전체 스킵 (바로 §3-A.1 답 처리로). 1개 이상이면 *해당 슬롯의 질문만* 객관식으로:

**질문 1 — 좋아하는 디자인 / 레퍼런스** (`AskUserQuestion`, `fav-references` 미충족 시만):

> 최근 *눈이 가서 저장해둔* 디자인이나 프로덕트 1-3개를 알려주세요. 링크/이름 다 좋아요. *왜 좋았는지*도 한 줄씩 함께 — 한 단어라도 좋아요.
>
> *왜 묻나요*: 이 답이 모든 화면 만들 때 "어떤 결로 갈지"의 출발점이 돼요.
>
> - **답하기** (자유 텍스트로 — 예: "Linear 랜딩 — 절제된 모노 결, Apple Vision Pro — 깊이감 있는 레이어, Pretendard 페이지 — 한글 타이포가 살아있음")
> - **잘 모르겠어요 / 나중에 채울게요** → 첫 작업 때 자연스레 묻고 채워져요

**질문 2 — 본인 대표 작업** (`AskUserQuestion`):

> 본인의 *최근 작업이나 포트폴리오* URL 1개만 알려주세요. Behance·Dribbble·개인 사이트·노션 페이지 다 좋아요.
>
> *왜 묻나요*: 평소 본인의 결이 모든 화면 만들 때 *시작점*이 돼요. 한 줄 답에서 톤·색·타이포·공간감을 모델이 다 끌어옵니다.
>
> - **URL 답하기** (자유 텍스트로)
> - **회사 작업이라 공개 URL은 없어요** → 평소 톤·결을 한 단어로 답하기 (선택)
> - **잘 모르겠어요 / 나중에** → 첫 작업 결과 보고 자연스레 누적

**왜 회피는 안 묻나** (시니어 디자이너 관점): "피하고 싶은 디자인"을 사전에 카탈로그로 묻는 건 디자이너에게 *AI 클리셰 목록*을 떠넘기는 부담. 회피는 사전 학습보다 *사후 누적*이 정확함 — 첫 결과 보고 "이거 좀…" 반응할 때 `aesthetic-guard`가 그 어휘 그대로 §피하고 싶은 디자인 슬롯에 누적. 추가로 작업 마무리 시 `aesthetic-guard`가 *능동적으로* 한 번 가볍게 묻는다(부정 신호 없어도). 기본 회피(흔한 폰트·이니셜 박스·Lorem ipsum 등)는 `aesthetic-guard` 본문에 *고정*으로 박혀 있어 따로 묻지 않아도 작동.

**답 처리** (§A 자유 응답 + §B 객관식 통합):
1. **§A에서 박힌 슬롯은 *덮지 X***. §B 객관식은 *빈 슬롯에만* 답을 박는다.
2. `fav-references` (좋아하는 디자인 / 레퍼런스): 자유 응답 원문 또는 객관식 답 → `~/.claude/CLAUDE.md` §디자이너 프로필 슬롯에 *원문 그대로*. 모델의 톤·색·타이포 추출은 *첫 작업 발동 시* 자동.
3. `portfolio` (본인 대표 작업): URL 답이면 슬롯에 박기. 톤 단어만 답이면 `tone-extracted` 슬롯으로. "나중에"면 placeholder 유지.
4. `avoidance` (피하고 싶은 디자인): §A 자유 응답에 회피 언급이 있으면 그 어휘 *원문 그대로* 박기. 그 외엔 *비워둠* — 사후 누적용.
5. 첨부 파일에서 추출한 결(폰트·색·구도)은 해당 슬롯 본문 끝에 *추출 요약 한 줄* 박기 — 디자이너가 *어떤 추론이 됐는지* 볼 수 있게.

**3개 질문 상한 점검** (글로벌 §6): 이 사이클의 *AskUserQuestion 호출 수* = §3 동의 1 + §3-A.1 §B 객관식 0~2 = 1~3회. 상한 안. **§A 자유 응답은 평문 메시지라 카운트 X** — 객관식 한도와 별도. 추가 질문 X.

### 4.1 사이클 시작 시점 기록 (export-handoff용)

`export-handoff`가 *이번 사이클에 추가/변경된 파일만* 인계 README에 정리할 때 기준점으로 쓸 SHA를 박아둔다. 박아두지 않으면 옛 mock·옛 import까지 끌려와 개발자에게 잘못된 인계가 가게 됨.

처리 절차:

1. **git 저장소가 아니면 스킵** (`git rev-parse --git-dir 2>/dev/null` 실패).
2. **commit 0개면 스킵** (`git rev-parse HEAD 2>/dev/null` 실패).
3. 그 외:
   - `mkdir -p ./.claude`
   - **이미 `.claude/.kd-session-base` 존재 시 덮어쓰지 X** (다른 흐름이 먼저 박았을 수 있음 — 처음 박는 1회만 책임).
   - 부재 시: `echo "$(git rev-parse HEAD)" > ./.claude/.kd-session-base`
4. **`.gitignore` 처리** (자동 추가):
   - `.gitignore`에 `.claude/.kd-session-base` 한 줄이 없으면 *append* (기존 줄 보존).
   - `.gitignore` 자체가 없으면 새로 만들어 그 한 줄만 박기.
   - `.claude/` 디렉토리 자체는 gitignore X.

이 SHA는 `export-handoff`가 `git diff $BASE_SHA..HEAD`로 *이번 사이클 변경분*만 추릴 때 기준점이 된다.

#### 6.2-A 마커만 삽입 (이전 버전 슬롯 정상화)

"마커만 부재" 분류된 슬롯들에 대해:

1. **백업 우선** — `cp ./CLAUDE.project.md ./CLAUDE.project.md.kd-migrate-<YYYYMMDD-HHmm>` (전역 수정이면 동일 패턴으로 `~/.claude/CLAUDE.md.kd-migrate-...`).
2. **마커 한 줄 삽입** — 사용자 파일에서 해당 한국어 헤더의 *정확 일치* 라인을 찾아, 그 라인 *바로 위*에 `<!-- kd:slot:ID -->\n` 한 줄 삽입.
3. **본문은 절대 손대지 X** — 사용자가 채운 모든 내용 보존.
4. *답을 묻지 않음* — 묵묵히 모든 "마커만 부재" 슬롯 처리 후 §6.2-B로.

#### 6.2-B 빈 골격 추가 ("둘 다 부재" 슬롯)

"둘 다 부재" 분류된 슬롯들을 §6.0 manifest의 카테고리에 따라 처리:

**카테고리: 답 필요** (`fav-references`, `portfolio`)
- 전역이 *지금 켜져 있는 경우에만* §6.3 인터뷰로 묻기.
- 전역이 *안 켜져 있으면* 이 카테고리 슬롯은 *건너뛴다* (전역에만 존재하는 슬롯이라 박을 자리가 없음).

**카테고리: 컨테이너 / 자동 누적 / 자동 추출 / 빈 골격**
- 묻지 않음. §6.0 manifest의 *해당 슬롯 빈 골격*을 마커 + 헤더 + placeholder 본문 형태로 append:
  ```
  
  <!-- kd:slot:ID -->
  ## 한국어 헤더 (or ### 으로 manifest 명시 그대로)
  > 안내 인용구 (manifest의 빈 골격 본문 그대로)
  
  - {{placeholder 1}}
  - {{placeholder 2}}
  ```
- 삽입 위치 결정:
  1. manifest 순서상 *그 슬롯 직전 슬롯*이 사용자 파일에 있으면 → 그 슬롯 끝 직후에 삽입.
  2. 직전 슬롯이 없으면 → *직후 슬롯* 직전에 삽입.
  3. 둘 다 없으면 → 사용자 파일 끝에 append.
- placeholder(`{{...}}`)는 그대로 — 디자이너가 자연어로 발화할 때 다른 Skill(`aesthetic-guard`/`design-system-guard` 등)이 채움.

#### 6.3 답 필요 슬롯 인터뷰

전역이 켜져 있고 §6.2-B 처리 대상에 *답 필요 슬롯*이 있을 때만 발동.

§3-A.1 인터뷰와 동일 카피 재활용 — *빠진 질문만* 묻기.

| 빠진 슬롯 | 질문 카피 | 옵션 |
|---|---|---|
| `fav-references` | "최근 *눈이 가서 저장해둔* 디자인이나 프로덕트 1-3개를 알려주세요. *왜 좋았는지*도 한 줄씩 함께." | 답하기 / 잘 모르겠어요 |
| `portfolio` | "본인의 *최근 작업이나 포트폴리오* URL 1개만 알려주세요." | URL 답하기 / 회사 작업이라 공개 URL은 없어요 / 잘 모르겠어요 |

질문 수 최대 2개. 글로벌 사이클 상한(3) 안.

답을 받으면 해당 슬롯 영역의 placeholder를 답으로 치환해 §6.2-B의 append 흐름으로.

#### 6.4 결과 응답 패턴 (필수 — 일반 "이미 시작됨" 응답 금지)

§6.1 분류 결과에 따라 *반드시* 아래 패턴 중 하나로 응답. 절대 만들지 않는 응답: "이미 시작되어 있어요 — 다시 시작하고 싶으면 /kdesigner:디자인끄기로..." 같은 *분기를 막는* 응답. 막지 말고 처리 결과를 말해야 한다.

**Case 1 — 처리 대상 0개 (모두 정상)**:

> 디자이너 모드 *이미 최신 상태*예요. 추가하거나 정리할 게 없어요.

**Case 2 — §6.2-A만 발생 (마커만 삽입, 새 슬롯 추가 0개)**:

> 디자이너 모드의 *내부 표시*를 최신으로 정리했어요. 화면에는 변화가 없고, 다음 업데이트가 더 깔끔하게 동작하도록 표시만 추가한 거예요. 이미 채워두신 내용은 그대로 보존했어요.

**Case 3 — §6.2-B만 발생 (새 슬롯 1개 이상 추가)**:

> 디자이너 모드를 *최신으로* 업데이트했어요. 이번에 추가된 자리:
> - {{슬롯 한국어 이름}} ({{전역/프로젝트}})
> - ...
>
> 이미 채워두신 내용은 그대로 보존했어요. 비어있는 새 자리는 *지금 채우지 X* — 첫 화면 만들 때 "어떤 분위기로?" 같은 *질문 1개*로 자연스레 채워져요.

**Case 4 — §6.2-A + §6.2-B 모두 발생**:

> 디자이너 모드를 *최신으로* 업데이트했어요. 이번에 추가된 자리:
> - {{슬롯 한국어 이름}} ({{전역/프로젝트}})
> - ...
>
> 그리고 기존 슬롯 몇 개의 *내부 표시*도 함께 정리했어요. 이미 채워두신 내용은 그대로 보존했어요. 비어있는 새 자리는 첫 화면 흐름에서 자연스레 채워져요.

각 슬롯의 한국어 이름 매핑(Case 3·4 응답 본문에 사용):

| ID | 한국어 이름 |
|---|---|
| `designer-profile` | 디자이너 프로필 |
| `fav-references` | 좋아하는 디자인 / 레퍼런스 |
| `portfolio` | 본인 대표 작업 |
| `avoidance` | 피하고 싶은 디자인 (자동 누적) |
| `tone-extracted` | 평소 톤 (자동 추출) |
| `design-philosophy` | 디자인 철학 |
| `project-tone` | 톤 한 단어 |
| `project-references` | 프로젝트 레퍼런스 |
| `project-avoidance` | 프로젝트 한정 회피 |
| `design-tokens` | 핵심 토큰 |
| `tech-stack` | 사용 스택 |
| `components` | 사용 가능한 컴포넌트 |
| `last-work` | 마지막 작업 |

#### 6.5 안전 장치

- **백업 우선** — 마이그레이션 *전*에 `~/.claude/CLAUDE.md`(전역 수정 시) 또는 `./CLAUDE.project.md`(프로젝트 수정 시)를 `.kd-migrate-<YYYYMMDD-HHmm>` 접미사로 백업. 백업 실패 시 마이그레이션 자체를 *중단* + 사용자에게 보고.
- **3-way 분류 후 손상 의심 시 즉시 중단** — §6.1 표의 "손상 의심" 행(같은 마커 ID 2번 이상, 같은 한국어 헤더 2번 이상 with 마커 0개) 한 건이라도 발견되면 *모든 자동 추가 중단* + 사용자에게 슬롯 ID 명시해 보고. 디자이너가 직접 본 후 다시 호출.
- **마이그레이션 후 검증** — `Read`로 다시 읽어 *모든 슬롯 ID가 정확히 1번씩* 존재하는지 확인. 0번 또는 2번 이상이면 백업 경로 안내 + 수동 정리 권유.
- **사용자 본문 보존 1순위** — §6.2-A는 마커 한 줄만 삽입하므로 본문에 *어떤 영향도 없다*. §6.2-B는 *진짜 빠진 슬롯*에만 동작하므로 사용자가 채운 영역과 충돌 없음. 둘 다 사용자 데이터를 *수정하지 않음*.

#### 6.6 §2~§5와의 관계

§6 마이그레이션 분기는 *이미 시작됨* 상태에서만 들어가므로:
- §2 (프로젝트 단위 활성화) — 스킵 (이미 활성화돼 있음)
- §3 (전역 활성화) — 전역이 *아직 안 켜졌으면* §3을 그대로 실행. 켜져 있으면 §6.1의 전역 슬롯 매칭만.
- §4 (검증) — 그대로 적용 (마커 검증).
- §4.1 (사이클 시작 SHA) — 이미 박혀 있으면 그대로, 없으면 박기.
- §5 (다음 흐름 안내) — 마이그레이션 후 디자이너의 *원래 의도*(첫 작업 시작? 이어서 작업?)는 모르므로, §6.4 응답 끝의 다음 행동 1개로 *자연어 권유* — "이제 어떤 화면을 더 만들어볼까요?" 또는 "지금까지 작업을 보여드릴까요?".

## 안전 장치

- **명시 동의 (전역만 해당)** — `~/.claude/CLAUDE.md` 수정 전 `AskUserQuestion` 1회. "무엇이 어떻게 바뀌는지" 1줄 설명 + "잘 모르겠어요" 옵션 필수(CLAUDE.md §5).
- **백업 우선** — 전역 추가 전 *반드시* 백업 (`cp`). 백업 실패 시 추가 X, 사용자에게 보고 후 중단.
- **마커로 격리** — 추가 내용을 `<!-- kd:designer-mode:start -->`~`end` 마커로 감싸 `/kdesigner:디자인끄기`가 정확히 그 부분만 제거할 수 있게.
- **재실행 안전** — `CLAUDE.project.md`/마커가 이미 있으면 덮지 않고 §6 분기.
- **비파괴** — 어떤 경우에도 사용자 파일을 *지우지* 않음. 추가·백업만(CLAUDE.md §3).
- **외부 도구 호출 금지** — `git`/`npm`/네트워크 호출 X. 이 명령은 *모드 켜기*만 담당.
