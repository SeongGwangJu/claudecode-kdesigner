## 목적
디자이너 친화 품질 리포트. 영어 lint 메시지가 아니라 *사람 말로* 문제 위치와 영향을 짚는다.

## 발동 조건

### 발동
- 점검·확인·검사 자연어
- `export-handoff` 직전 자동 호출 (인계 전 마지막 점검)

### 발동 X
- 코드 변경 의도 ("바꿔줘"/"고쳐줘") → 다른 Skill
- 빌드 검증만 → `auto-validate`

## 처리 흐름

### 1. 점검 대상 파일 선정
직전 변경된 파일 + 컴포넌트 인덱스(`CLAUDE.project.md`) 참고. 사용자가 "이 화면" 명시하면 해당 파일만.

### 2. 4개 체크리스트 실행

#### 2.1 반응형 누락
정적 분석으로 추정 (정확한 측정엔 한계, 의심 패턴만 지목):

- 고정 픽셀 width 박힘 (`width: 800px`, `w-[800px]`) — 모바일에서 잘릴 가능성
- Tailwind breakpoint 접두사(`sm:`/`md:`/`lg:`) 부재 + 큰 padding/font 값 — 모바일 미고려 의심
- `<table>` 내부에 모바일 대응 클래스 없음

#### 2.2 색 대비 (WCAG AA 추정)
정확한 명도 측정은 빌드 시점에서 어렵고, *의심 패턴*만 지목:

- 회색 톤 위 회색 톤 (`text-gray-400` on `bg-gray-200` 류)
- 어두운 배경 위 어두운 텍스트 / 밝은 배경 위 밝은 텍스트
- 토큰 정의된 `muted` 류 색 위에 `muted-foreground`가 아닌 `muted` 동족 색

리포트엔 *어디서, 왜 의심되는지*만 — "이 부분 글씨가 배경과 너무 비슷해서 안 보일 수 있어요".

#### 2.3 `alt` 누락
- `<img>` / `<Image>` 태그 중 `alt` 속성 부재 또는 `alt=""` (장식용 외 의도)
- `Grep`으로 일괄 스캔

#### 2.4 디자인 토큰 어긋남 (design-system-guard와 분담)
- 인라인 hex/rgb 색 (`#fff`, `rgb(...)`) — 토큰 외 직접 색
- 임의 spacing 값 (`p-[13px]`, `m-[7px]`) — 토큰 그리드 외
- 발견 시 *분류만* 보고하고 수정 분기는 `design-system-guard`로 위임 안내

### 3. 리포트 형식 — 디자이너 친화

심각도 3단계 + 자연어 위치 표기:

```
🔴 색 대비 약함
   메인 화면 카드 안 글씨 (`components/Card.tsx`)
   → 배경과 너무 비슷해서 잘 안 보일 수 있어요. 텍스트 색을 한 단계 더 진하게 어떨까요?

🟡 모바일 반응형 빠짐
   상단 헤더 (`components/Header.tsx`)
   → 800px 고정으로 박혀 있어요. 모바일에서 가로 스크롤이 생길 수 있어요.

🟢 접근성·토큰 점검은 모두 통과했어요
```

규칙:
- 빨강 = 사용자 노출 단계 가기 전에 고치는 게 좋음 / 노랑 = 환경에 따라 문제 / 초록 = OK
- 영어 메시지 단독 노출 X — 파일명·식별자만 백틱으로
- 발견 0건이어도 리포트는 보낸다 (안심)

### 4. 다음 행동 분기
리포트 끝에서 1개만 제안:

| 발견 결과 | 제안 |
|---|---|
| 빨강 1개 이상 | "이 부분만 같이 고쳐볼까요?" |
| 노랑만 | "지금 고칠지, 일단 두고 인계할지 골라드릴까요?" |
| 모두 초록 | "이제 개발자한테 넘길 정리 시작할까요?" |

## Subagent 위임
